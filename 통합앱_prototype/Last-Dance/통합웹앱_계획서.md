# 스마트스토어 통합 웹앱 — 계획 & 검증 문서 (v1)

> 목표: **키워드 분석 → 상세페이지 텍스트 생성 → 이미지 생성** 세 기능을 하나의 웹앱으로 묶는다.
> 방식: **2단계(① 프로토타입 → ② 실제 키 연동)**, 단 착수 전 계획·검증을 철저히 한다.
> 작성: Cowork / 검토 후 승인 시 Phase 1 착수.

---

## 0. 한 줄 결론
필요한 API 키가 `.env`에 **모두 준비**돼 있고, 키워드 분석기가 **이미 Streamlit 웹앱**이라, 세 기능을 하나의 Streamlit 멀티탭 앱으로 통합하는 것이 **기술적으로 충분히 가능**하다. 권장 경로는 "기존 키워드 앱을 뼈대로 삼아 상세페이지·이미지 탭을 붙이는" 방식.

## 1. 확인된 자산 (사실 기반)

### 1-1. `.env` 에 설정된 키 (값은 미열람, 이름만 확인)
| 키 | 용도 | 어느 기능에 쓰나 |
|----|------|------------------|
| `NAVER_CLIENT_ID` / `NAVER_CLIENT_SECRET` | 네이버 검색/쇼핑/데이터랩 OpenAPI | 키워드·쇼핑 분석 |
| `NAVER_AD_CUSTOMER_ID` / `NAVER_AD_ACCESS_LICENSE` / `NAVER_AD_SECRET_KEY` | 네이버 **검색광고 API**(검색량·경쟁도) | 키워드 분석 핵심 |
| `GEMINI_API_KEY` | Google Gemini 이미지 생성/편집 | 이미지 생성 |
| `REMOVE_BG_API_KEY` | remove.bg 배경 제거 | 이미지 후처리 |
| `ANTHROPIC_API_KEY` / `OPENAI_API_KEY` | LLM(카피 생성) | 상세페이지 텍스트 |
| `NAVER_COMMERCE_CLIENT_ID` / `NAVER_COMMERCE_CLIENT_SECRET` | 네이버 커머스(스마트스토어) API | (보너스) 업로드 |
| `DOMEGGOOK_API_KEY` | 도매꾹 오픈API | (보너스) 소싱 |

> ⚠️ 보안: `.env`에 **실제 비밀키**가 들어 있음. 코드에서 런타임으로만 읽고 화면/로그/깃에 노출 금지. `.gitignore`에 `.env` 포함 필수.

### 1-2. 모듈별 현황
| 모듈 | 폴더 | 형태 | 상태 |
|------|------|------|------|
| 키워드 분석 | `네이버_키워드_분석기/app.py` | **Streamlit 웹앱 (2,600줄)** | 검색광고/쇼핑/데이터랩 API 연동 코드 **이미 존재** |
| 이미지 생성 | `나노바나나_이미지생성_MCP` | TS MCP (`@google/genai`) | Gemini 호출 로직 참고용. Python으로 재구현 권장 |
| 상세페이지 카피 | `크로스보더_이커머스_스킬/amazon/...listing-copywriter.md` | 프롬프트(스킬) | LLM에 넣을 지시문. 한국 스마트스토어용으로 변형 필요 |
| (보너스) 소싱 | `1688_소싱_CLI` (TS) + `DOMEGGOOK_API_KEY` | CLI/API | 후속 단계 |
| (보너스) 업로드 | `NAVER_COMMERCE_*` | API | 후속 단계 |

### 1-3. 기존 키워드 앱이 이미 하는 일 (app.py 확인)
- 네이버 **검색광고 keywordstool** 로 연관키워드·월검색량·경쟁도 조회 (`get_related_keywords`, `make_searchad_signature`)
- 네이버 **쇼핑 API**로 상품수/경쟁 강도 (`get_shopping_info`, `search_shopping_page`)
- **데이터랩** 추세, 상품 순위 추적, 프리셋 저장(쿠키) 등
- → 키워드 기능은 **신규 개발이 아니라 재사용**이 가능.

## 2. 목표 사용자 흐름 (UX)
```
[카테고리/키워드 입력]
   → ① 키워드 자동 분석 (검색량·경쟁도·상품화 점수)
   → ② 사용자가 키워드 선택
   → ③ 선택 키워드 기반 상세페이지 카피 생성 (제목·셀링포인트·상세·태그)
   → ④ 이미지 생성 (메인컷/연출컷, 배경제거 옵션)
   → ⑤ 미리보기 + 다운로드(HTML/이미지/엑셀)
   (→ 보너스: 도매꾹 소싱 후보 / 네이버 커머스 업로드 패키지)
```

---

## 3. 권장 아키텍처
- **프레임워크: Streamlit (Python)** — 이유: 키워드 앱이 이미 Streamlit, 로컬에서 `streamlit run`으로 즉시 웹앱, `.env` 그대로 사용, 통합 비용 최소.
- **구조(멀티탭 단일 앱)**
  ```
  app.py (메인)
  ├─ tab_keyword.py   ← 기존 app.py 로직 이식/재사용
  ├─ tab_detail.py    ← LLM 카피 생성 (Anthropic/OpenAI)
  ├─ tab_image.py     ← Gemini 이미지 + remove.bg
  ├─ core/keyword_api.py / llm.py / image.py  ← API 래퍼
  └─ .env (키)        ← python-dotenv 로 로드
  ```
- **데이터 전달**: `st.session_state`로 탭 간 공유(선택 키워드 → 카피 → 이미지 프롬프트 자동 채움).

## 4. 2단계 로드맵

### Phase 1 — 프로토타입 (키 없이/최소 호출, UX·뼈대 검증)
목표: 전체 화면 흐름과 데이터 전달이 동작함을 증명.
- [ ] P1-0 환경 검증: Python/Streamlit 설치, `.env` 로드, 기존 app.py 실행 가능 여부 확인
- [ ] P1-1 통합 앱 골격(3탭) + session_state 연결
- [ ] P1-2 키워드 탭: **모의 데이터**로 표/점수 렌더 (기존 함수 시그니처에 맞춤)
- [ ] P1-3 상세페이지 탭: 입력→카피 생성 자리(모의 출력)
- [ ] P1-4 이미지 탭: 프롬프트 입력→플레이스홀더 이미지
- [ ] P1-5 결과 미리보기/다운로드(HTML·엑셀) 동작
- 산출물: 로컬에서 도는 `app.py`(모의), 실행법 README
- 수용 기준: 입력 한 번으로 ①~⑤ 화면이 끊김 없이 이어짐

### Phase 2 — 실제 키 연동 (실데이터)
목표: 모의 → 실제 API로 교체, 실사용 가능.
- [ ] P2-1 키워드: 검색광고/쇼핑 API 실연동(기존 app.py 함수 재사용) + 상품화 점수식 확정
- [ ] P2-2 상세페이지: Anthropic/OpenAI 실호출 + 한국 스마트스토어용 프롬프트 튜닝
- [ ] P2-3 이미지: Gemini 실호출 + remove.bg 옵션
- [ ] P2-4 오류/한도/비용 가드(타임아웃·재시도·호출량 표시)
- [ ] P2-5 통합 E2E 테스트 + 사용설명서
- (보너스) P2-6 도매꾹 소싱 탭 / P2-7 네이버 커머스 업로드 패키지
- 수용 기준: 실제 키워드 1건으로 분석→카피→이미지까지 실데이터로 산출

## 5. API별 상세·비용·리스크 (검증 대상)
| API | 인증 방식 | 비용/한도 | 검증할 점 |
|-----|----------|-----------|-----------|
| 네이버 검색광고 | HMAC 서명(타임스탬프+키) | 무료(계정 보유), 호출 한도 있음 | 서명 로직(기존 코드에 있음) 실동작, 한도 초과 처리 |
| 네이버 쇼핑/데이터랩 | Client ID/Secret | 일일 호출 제한 | 제한 도달 시 폴백 |
| Gemini 이미지 | API Key | **유료**(이미지당 과금) | 모델명·요금·생성 속도, 실패 재시도 |
| remove.bg | API Key | 크레딧제(유료) | 무료 크레딧 한도 |
| Anthropic/OpenAI | API Key | **유료**(토큰) | 1건당 예상 비용, 길이 제한 |
| 네이버 커머스 | OAuth2 | 심사·약관 | 자동 업로드 약관 적합성(아래 6번) |

## 6. 레드팀 — "이게 왜 안 될 수 있나" (가정 → 실패조건 → 가장 싼 검증)
1. **가정: 기존 app.py가 그대로 돈다.** 
   - 실패조건: 의존 라이브러리/쿠키매니저/세션 구조가 통합 앱과 충돌. 
   - 가장 싼 검증: Phase 1-0에서 app.py 단독 `streamlit run` 1회 실행. ← **최우선**
2. **가정: 검색광고 API 서명이 유효하다.** 
   - 실패조건: 키 만료·권한 부족·서명 불일치. 
   - 검증: keywordstool에 키워드 1개 호출해 200 응답 확인.
3. **가정: Gemini 키로 이미지가 생성된다.** 
   - 실패조건: 모델 접근 권한 없음·리전 제한·과금 미설정. 
   - 검증: 텍스트→이미지 1장 최소 호출.
4. **가정: 비용이 감당 가능하다.** 
   - 실패조건: 이미지/LLM 과금이 예상보다 큼. 
   - 검증: 1회 풀 파이프라인 실비용 측정 후 단가표 작성.
5. **가정: 네이버 커머스 자동 업로드가 약관상 허용.** 
   - 실패조건: API 권한 미승인·자동등록 제약. 
   - 검증: 보너스 단계로 분리, 우선 "업로드 패키지(파일) 생성"까지만.
6. **가정: AI 생성 이미지가 상품 실물과 일치.** 
   - 실패조건: 실물 불일치 → 과장광고 리스크. 
   - 대응: 제품 본컷은 실사, AI는 배경/연출/인포그래픽으로 한정(정책).

## 7. 검증(테스트) 계획
- **모듈 단위**: 각 API 래퍼에 "1건 호출" 스모크 테스트(키워드 1개, 카피 1개, 이미지 1장).
- **통합 E2E**: 샘플 상품(예: 로카티)로 입력→분석→선택→카피→이미지→다운로드 1회 완주.
- **오류 케이스**: 키 누락/한도 초과/빈 입력/네트워크 실패 시 사용자에게 명확한 메시지.
- **회귀**: Phase 2 교체 후에도 Phase 1 화면 흐름 유지 확인.
- **품질 게이트**: 각 Phase 종료 시 체크리스트 통과 + 사용자 확인 후 다음 단계.

## 8. 결정 필요 사항 (사용자 확인)
1. **카피 LLM**: Anthropic(Claude) vs OpenAI(GPT) 중 기본값? (둘 다 키 있음)
2. **이미지 범위**: AI는 연출/배경만 / 제품 본컷은 실사 — 이 원칙으로 OK?
3. **보너스 포함 여부**: 도매꾹 소싱·네이버 커머스 업로드를 1차 범위에 넣을지, 후속으로 뺄지.
4. **실행 환경**: 본인 PC 로컬 실행(권장) vs 외부 배포(Vercel 등 — Streamlit은 별도 호스팅 필요).
5. **상세페이지 출력형식**: HTML 미리보기 / 이미지 합성본 / 텍스트(카피) 중 우선순위.
